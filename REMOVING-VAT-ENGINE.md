# Removing the VAT Filing Engine

The VAT Filing Engine is a strictly separate module. Nothing outside it depends on it —
its only outward call is `duties.apply_completion()` (a core function that exists
regardless of this module). Removal is deletion plus unhooking four marked
registration points.

## Instant off-switch (no code changes)

Set the environment variable and restart the API:

```
VAT_ENGINE_ENABLED=false
```

Every `/vat-engine/*` endpoint returns 404 and the frontend hides all VAT UI (the nav
button probes `/vat-engine/status` and renders nothing on 404). Data stays in place.

## Full removal

### 1. Delete the module files

```
api/app/routers/vat_engine.py
src/vat-engine/                      (entire folder)
api/tests/test_vat_engine.py
api/alembic/versions/c9e5a12b8d47_vat_engine_tables.py   (see step 4 first)
REMOVING-VAT-ENGINE.md               (this file)
```

### 2. Remove the marked registration lines

- `api/app/main.py` — the two lines under the `# VAT-ENGINE` comment
  (`from .routers import vat_engine` and `app.include_router(vat_engine.router)`).
- `api/alembic/env.py` — the `try: from app.routers import vat_engine` block.
- `src/app-production.jsx` — three lines marked `VAT-ENGINE (removable)`:
  the import of `./vat-engine/index.jsx`, the `<VatEngineNav …/>` line in the nav,
  and the `{route.screen === "vat" && <VatEngineScreen />}` line in the screen switch.

### 3. Remove the dependencies (optional)

`pandas` and `openpyxl` in `api/requirements.txt` are used only by this module.

### 4. Drop the tables

Preferred: **before** deleting the migration file, downgrade past it:

```bash
cd api && alembic downgrade b7c3d84f1a52
```

Or drop directly (then delete the migration file and
`alembic stamp` accordingly / edit `alembic_version` if it points at `c9e5a12b8d47`):

```sql
DROP TABLE IF EXISTS vat_client_requests, vat_filing_events, vat_filing_items, vat_filings;
```

Files uploaded/generated by the module live in the shared `files` table
(`entity = 'vat_filing'`, plus registry documents under `entity = 'client'`) and in blob
storage; they are inert and can stay, or be deleted by those entity values.

## Generic pieces that intentionally STAY (not module-specific)

- `api.js` `getBlob` — generic authenticated binary download.
- `clients.py` "Client registry" document source (`files` rows with `entity='client'`) —
  a generic capability any feature can use.
- `duties.apply_completion()` — the extracted core of duty completion, used by the
  `/duties/{id}/complete` endpoint itself.
